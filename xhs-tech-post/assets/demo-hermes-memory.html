5 个隐藏成本
  • cosine sim 一刀切 · 时序丢光（早晨 vs 上周一视同仁）
  • top-k 易塞满 16K context（k=20×800tok 就吃光）
  • 偏好被闲聊淹没（"喜欢 Python" 埋在 100 条 "thanks"）
  • 长尾任务从未召回（sim 偏向高频 query）
  • 偏好 update 越积越多（旧条目不删 → 召回噪声）
代价 · 一种检索方式撑不起所有情境

agent memory · note 01
什么时候该 抄 这套
三层成本不低 · 适用场景要看清
适用
  • 长期对话连续性是核心需求（个人助手、长任务 agent）
  • 能容忍 1-2 周 episodic 冷启动期
  • LLM 调用预算充足（周期 nudge 是大头）
不适用
  • 单次完成型任务（short-term 一层就够）
  • 强 stateless 需求（每次重置）
  • 资源紧张 / 不能多 LLM 调用

post body (正文)

看了 NousResearch/hermes-agent 的 memory 实现挺有意思——市面上大多数 agent 的"记忆"还是 vector store + cosine sim 一套打天下。Hermes 干脆拆成 short-term / long-term / episodic 三层，每层用最匹配的存储和召回方式。

看起来复杂，但在长对话连续性场景下精度提升肉眼可见。代价是三套基础设施 + episodic 前 1-2 周的冷启动期。
